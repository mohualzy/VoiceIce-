
# 读前须知

## 什么是Git?

简单来说，**Git 是一个分布式版本控制系统**。

为了理解这个定义，我们把它拆解开：

1. **版本控制**：
    
    - 想象一下你在写一篇重要的论文或一份代码。你肯定会不断地修改和保存。如果没有版本控制，你的文件夹里可能会充满了“论文_v1.doc”、“论文_v2_修改版.doc”、“论文_最终版.doc”、“论文_真的最终版.doc”这样的文件，非常混乱。
        
    - **Git 就是来解决这个问题的**。它就像一个“时光机器”，可以记录你对文件（通常是代码）的每一次修改。你可以随时回到任何一个历史版本，看看某一行代码是谁、在什么时候、为什么修改的。
        
2. **系统**：
    
    - 这意味着 Git 是一套成熟、强大的工具和协议，它定义了如何创建、存储、追踪和合并这些版本。
        
3. **分布式**（这是 Git 最核心、最革命性的特点）：
    
    - 在 Git 出现之前，很多版本控制系统是“集中式”的（如 SVN）。它们有一个中央服务器，存放着所有版本历史。每个人都需要连接到这个服务器才能获取最新的代码或提交自己的修改。如果服务器坏了，大家就都没法工作了。
        
    - **Git 是分布式的**：这意味着**每个人的电脑上都有一个完整的代码仓库（包括全部的历史记录）**。你可以在自己的电脑上独立地进行版本控制，比如提交代码、查看历史、创建分支，而完全不需要网络连接。
        
    - 当然，为了团队协作，通常会有一个大家公认的“中央”仓库（比如放在 GitHub 或 GitLab 上），但这个仓库在地位上和每个开发者的本地仓库是平等的，它并不是唯一的“真理之源”。
        

**一个生动的比喻：**  
把项目（比如一个软件）想象成一个游戏。

- **Git** 就是这个游戏的**存档系统**。
    
- 每一次 `git commit`（提交）就是一次**手动存档**，并附上存档说明（比如“打败了第一个BOSS”）。
    
- 你可以随时**读档**，回到任何一个之前的存档点。
    
- **分支**就像是游戏的**多周目**或**平行世界**，你可以在一个存档点上发展出完全不同的剧情，最后再决定把哪个剧情合并到主线上。
    
- **分布式**意味着你和你的朋友每人都有这个游戏的完整客户端和所有存档，可以独立玩，也可以互相交换存档。

## 为什么要用Git?

使用 Git 的理由非常多，无论是个人项目还是团队协作，它都带来了翻天覆地的变化。

#### 对个人开发者：

1. **完美的版本回溯能力**
    
    - **场景**：你修改了一大堆代码，结果程序跑不起来了，你也不记得改了哪里。
        
    - **Git 的解决**：一键 `git reset` 或 `git checkout`，立刻回到修改前的稳定状态。你再也不用担心“改坏代码”了。
        
2. **清晰的项目修改历史**
    
    - Git 要求每次提交都必须写日志。这形成了一个非常清晰的项目演进历史。未来你（或其他人）回头看时，能清楚地知道每一次修改的意图。
        
3. **强大的分支功能**
    
    - 你可以为每一个新功能、每一个实验性的想法创建一个独立的“分支”。在这个分支上随便折腾，完全不会影响主线（master/main）的稳定。等开发测试完毕，再合并回主线。这极大地鼓励了创新和试验。
        

#### 对团队协作：

这是 Git 大放异彩的地方。

1. **高效的并行开发**
    
    - 多个开发者可以在同一个项目的不同功能分支上同时工作，互不干扰。比如，张三在开发登录功能，李四在开发支付功能，他们可以同时进行，最后再将代码合并。
        
2. **解决代码冲突**
    
    - 当两个人修改了同一处代码时，Git 会清晰地标记出冲突，并引导开发者进行沟通和解决。这比过去通过邮件或U盘来回发送代码文件要科学和高效得多。
        
3. **责任明晰**
    
    - Git 记录了每一行代码的修改者。如果某处代码出了问题，可以快速定位到责任人，方便追查问题和代码审查。
        
4. **备份与容灾**
    
    - 因为每个开发者的电脑上都有完整的仓库，这相当于项目有了无数个备份。即使中央服务器（如 GitHub）彻底宕机，也可以从任何一个团队成员的本地仓库恢复整个项目，损失极小。
        

#### 与现代开发流程的结合：

以 **GitHub**、**GitLab** 为代表的代码托管平台，将 Git 的强大功能与 Web 服务结合，催生了现代软件开发的标准流程：

- **代码审查**：通过 Pull Request 或 Merge Request，让团队成员在合并代码前进行审查和讨论。
    
- **持续集成/持续部署**：当代码被推送到仓库时，自动触发测试、构建和部署流程。
    
- **项目管理**：与 Issue、Wiki、项目管理看板等工具无缝集成。
